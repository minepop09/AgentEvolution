# 讨论纪要

> 日期：2026-05-19

## 我提出的结构化架构

### 核心观点

用户需求的本质是：构建一个**可移植、可进化的问题处理操作系统**，类似人类的问题处理方法论 + 工具箱。

### 架构分层

```
Agent Soul
├── 问题处理OS（7个模块流水线）
│   ├── 问题预处理 → 问题认知 → 问题分析
│   └── 工具审核 → 工具安装
│   └── Todo计划模块（贯穿全程）
└── 知识库（可进化，跨Session持久）
```

---

## 关键讨论点

### 1. "问题认知模块"的边界最模糊

**问题**：模块的触发条件是"没有专业性认知"——但这个判断谁来做？

**三个选项**：
- Agent 自己判断（它怎么知道自己不懂？）
- 外部触发（用户指定领域）
- 先走一遍问题预处理模块，输出是"需要学"才触发

**结论**：建议问题预处理模块输出两条路：
- `无需学` → 直接进问题分析
- `需要学` → 进问题认知模块 → 学完 → 进问题分析

---

### 2. "工具审核模块"是最难自动化的部分

**问题**：找工具和判断最优解是两件事
- 找：有成熟方案（find-skills、RSS监控等）
- 判断最优解：目前没有好方法

**结论**：建议设计成**半自动**——AI枚举候选工具，给出对比矩阵，人/Agent选择

---

### 3. "自主进化"需要定义清楚进化什么

**问题**：根据 Agent 类型进化，进化的载体是什么？
- 知识库内容增加？（只是积累，不是进化）
- 工具选择偏好改变？（调整优先级）
- Skill 组合方式改变？（形成特定领域的 pipeline）

**结论**：进化的单位是 **Skill 编排顺序** + **工具权重**，不是知识本身

---

### 4. 本地知识库选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| Obsidian | 可视化，生态成熟 | Markdown文件，查询弱 |
| SQLite + 全文搜索 | 简单，查询快 | 无语义 |
| Qdrant/Milvus | 向量检索，语义搜索 | 需要额外服务 |
| txt2tags 纯文件 | 零依赖 | 查询全靠grep |

**结论**：先用**纯文件方案**（subject-creator 架构已支持），等需求明确再迁移向量库

---

## ✅ 已确认结论

| 问题 | 结论 |
|------|------|
| 问题认知模块边界 | 三层知识探测机制（置信度自评 → 知识库验证 → 主动探测） |
| 工具审核模块 | 按网络评分取前2，SlowMist审计，Low不装，失败换下一个 |
| 进化单位 | Skill编排顺序 + 工具权重 |
| 知识库选型 | 纯文件方案（Markdown），后续迁移向量库 |
| 知识库查询 | 主动查 |
| 最小版本 | 只做7个模块 |
| 目标Agent | OpenClaw 优先接入 |
